00 — Development Strategy: Local First, Cloud When Ready

Build everything on your laptop. Test everything on your laptop. Deploy to cloud only when the product is ready for real contractors. This saves money AND makes iteration faster.

The Two-Phase Approach

Phase A: Local Development (Weeks 1-4) — Everything runs on your laptop. PostgreSQL installed locally. FastAPI runs with uvicorn. Next.js runs with npm run dev. Photos saved to a local folder. The only external service is Gemini API (free tier — 1,000 requests/day). Total cost: $0/month.

Phase B: Cloud Deployment (Week 5+) — When ready for beta testers, deploy to managed services. Every service starts on free tier. Total cost: $0/month until revenue starts.

01 — System Architecture Overview

Hybrid managed stack. Google Cloud Run for compute (pay-per-request, scales to zero), specialized free-tier services for everything else. $0/month at launch, scales to $75/month at 500 users.

Why This Stack, Not AWS/GCP-Everything/Azure

We evaluated three approaches: (1) All-AWS (EC2 + RDS + S3 + SES + Cognito), (2) All-Google-Cloud (Cloud Run + Cloud SQL + Cloud Storage + Firebase Auth), (3) Hybrid — best-of-breed free tier per service. Hybrid wins decisively for a bootstrapped solo founder. Here's why:

  • AWS/Azure — Free tier expires after 12 months on most services. EC2/RDS run 24/7 even with zero traffic = paying for idle time. Setting up VPC, security groups, IAM policies, and load balancers takes 2-3 days of DevOps that should be spent building the product. Unpredictable bills (a misconfigured NAT gateway = surprise $30/month).
  • All-Google-Cloud — Cloud Run is excellent (free tier never expires, pay-per-request). But Cloud SQL has NO free tier (starts ~$7/month even idle). Cloud Storage charges $0.12/GB egress (photos served to homeowners = real cost). Firebase Auth lacks multi-tenant organizations out of the box.
  • Hybrid (our choice) — Cloud Run for compute (best serverless free tier + pay-per-request at scale). Neon for PostgreSQL (free forever, scale-to-zero). R2 for photos (zero egress fees). Clerk for auth (multi-tenant built-in). Each service is best-in-class for its job AND has the best free tier.

Request Flow — A Single Estimate (End to End)

This is what happens when Mike photographs Sarah's AC unit. Every step mapped to an API call:

  1. Mike opens app → GET /api/jobs/today (from Jobber webhook or manual) → Home screen renders
  2. Mike taps "New Assessment" → GET /api/properties/search?address=4215+Oakwood → Returns property history (or empty for new address)
  3. Mike takes 2 photos → POST /api/assessments/upload (multipart, 2 images) → Photos uploaded to R2, returns assessment_id
  4. Mike taps "Analyze" → POST /api/assessments/{id}/analyze → Cloud Run calls Gemini Vision API → Returns equipment identification + condition + issues
  5. App shows Intelligence Results → Data from step 4. Backend also runs GET /api/equipment/lookup?brand=Carrier&model=24ACC636 to enrich with lifecycle data
  6. Mike taps "Generate Estimate" → POST /api/estimates/generate (assessment_id + job_type + adjustments) → Backend calculates Good/Better/Best from pricing DB → Returns estimate with line items
  7. Mike reviews, adjusts markup slider → PATCH /api/estimates/{id} (updated line items / markup)
  8. Mike taps "Generate Documents" → POST /api/estimates/{id}/documents → Backend generates contractor PDF + homeowner report HTML → Stores in R2 → Returns URLs
  9. Mike shows homeowner report on phone → Loads report.scopesnap.com/{company_slug}/rpt-{id} (public, no auth)
  10. Mike taps "Send to Customer" → POST /api/estimates/{id}/send (phone + email) → Sends SMS via Twilio + email via Resend → Schedules follow-ups (24h, 48h, 7d)
  11. Sarah opens report, approves → POST /api/reports/{token}/approve (selected_option + signature) → Triggers Stripe payment link
  12. Sarah pays deposit → Stripe webhook → POST /api/webhooks/stripe → Updates estimate status to "approved + deposit paid"
  13. Estimate saved to history → POST /api/assessments/{id}/complete → Updates property record, equipment record, tech accuracy baseline
CRITICAL DESIGN RULE: Steps 4 and 6 are the only steps that touch AI APIs. Everything else is database queries, file generation, and HTTP calls to third-party services. The AI cost is exactly 1 vision call + 1 tiny LLM call per estimate. If you find yourself making more AI calls than that, you're doing it wrong.

03 — API Endpoints

FastAPI. All endpoints return JSON. Auth via Clerk JWT (Bearer token). Company ID extracted from JWT claims.

Authentication Middleware

# Every request passes through this
async def get_current_user(authorization: str = Header(...)):
    token = authorization.replace("Bearer ", "")
    claims = clerk.verify_token(token)
    user = await db.users.get(clerk_user_id=claims["sub"])

Prompt Testing & Improvement Protocol

  1. Build a labeled test set: Collect 50+ equipment photos from Upwork HVAC clients, Google Images, contractor forums. Label each with: brand, model, condition, visible issues. This is the ground truth.
  2. Run against both models: Gemini Flash and GPT-4o. Score each on: brand accuracy, model number accuracy, condition accuracy, annotation quality.
  3. Track metrics: Store every AI call result + tech overrides in assessments.ai_analysis and assessments.tech_overrides. When a tech corrects the AI, that's labeled training data for free.
  4. Monthly prompt review: Look at the 20 lowest-confidence assessments and the 20 most-overridden assessments. What patterns does the AI miss? Update the prompt.
  5. Model-specific tuning: Gemini Flash and GPT-4o respond differently to the same prompt. Maintain model-specific prompt variants if needed (e.g., Gemini needs more explicit JSON formatting instructions).
THE FEEDBACK LOOP: Every tech override makes the AI smarter. Not through fine-tuning (too expensive and fragile) — through better prompts, better equipment DB matching, and better known-issue patterns. At 10,000 assessments, we have 10,000 labeled examples of what the AI got right and wrong. That's a competitive advantage no new entrant can replicate.

05 — Estimate Generation Pipeline

Zero AI cost. This is pure math + database lookups + template rendering.

Pipeline Steps

  1. Get AI analysis results from assessment (already stored in DB)
  2. Determine job types: Map condition → recommended job types. If coil is moderate_corrosion → offer [clean_treat, coil_replacement, full_system]. This mapping is a simple lookup table, not an AI call.
  3. Look up pricing: For each job type, query pricing_rules with cascade: company → region → national.
  4. Calculate line items: Parts cost + (labor_hours × labor_rate) + permit + refrigerant + disposal. Pure arithmetic.
  5. Apply markup: Company default or tech override. (unit_cost × (1 + markup_percent / 100)).
  6. Calculate energy savings: If recommending system replacement, compare current SEER vs new SEER. Formula: annual_savings = annual_cooling_cost × (1 - current_seer / new_seer). Use EIA average for annual cooling cost by state.
  7. Check rebate eligibility: Query DSIRE database (or our cached copy) for state/utility rebates on high-efficiency equipment. Include in "best" tier.
  8. Build options array: 2-3 tiers with all line items, totals, savings, and descriptions.
  9. Calculate 5-year total cost: For each option: upfront + (annual_energy × 5) + likely_future_repairs. This powers the comparison chart in the homeowner report.

Serial Number Decoding (Example: Carrier)

# Carrier serial: First 2 chars = year code, chars 3-4 = week
# Example: "3920E12345" → year code "39" → 2020, week "20"
CARRIER_YEAR_MAP = {
    "29": 2010, "30": 2011, "31": 2012,
    "32": 2013, "33": 2014, "34": 2015,
    "35": 2016, "36": 2017, "37": 2018,
    "38": 2019, "39": 2020, "40": 2021,
    "41": 2022, "42": 2023, "43": 2024,
    "44": 2025, "45": 2026,
}

def decode_carrier_serial(serial: str) -> dict:
    year_code = serial[:2]
    week = serial[2:4]
    return {
        "year": CARRIER_YEAR_MAP.get(year_code),
        "week": int(week) if week.isdigit() else None
    }
# Each brand has its own decode pattern — stored in equipment_models.serial_decode_pattern
