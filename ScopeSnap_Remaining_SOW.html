Phase 1 — Navigation & Core Field UX
RW-01 Bottom Navigation Bar — Mobile Primary Navigation
~3 hrs P0 — Field Blocker

Why This Exists

HVAC techs use this app one-handed while holding tools. The current sidebar (hamburger → slide-out) is a two-tap, two-handed operation. The prototype shows a 5-tab bottom bar that techs can reach with their thumb without looking. This is the single biggest usability gap between the current build and something a tech can actually use in the field.

Steve Jobs rule: The most important actions must be reachable with one thumb, in under one second, without looking down. On a phone, that means a bottom tab bar — not a sidebar.

Phase 2 — Assessment Flow Fix
RW-02 Assessment Camera Flow — End-to-End Verification & Fix
~2–3 hrs P0 — Field Blocker

Why This Exists

The assessment page (/assess) is the entire product. If the photo capture → AI analysis → estimate creation chain fails silently or shows a blank screen at any step, the tech loses the job. This WP audits and hardens every step of the chain, and ensures the loading states and error messages match the prototype exactly.

Steve Jobs rule: Every screen in a flow must answer "what is happening right now?" The tech should never see a blank screen or a spinner with no label. Every state has a message.

The 5 Phases to Verify (in order)

Open app/(app)/assess/page.tsx. The flow has these phases: capture → uploading → analyzing → results → estimating

Phase 1 — Capture Screen

Matches prototype screen "Screen 2 — Capture". Must show:

  • Property address input with autocomplete from existing properties in the DB
  • Homeowner name + phone fields
  • Photo grid showing selected photos (up to 5)
  • Two ways to add photos: (A) File picker for web, (B) Camera button that opens capture="environment" for mobile
  • A single prominent CTA: "Analyze Equipment →" button (brand-green, full-width, 56px tall)

Issue to fix: Verify the camera input uses accept="image/*" capture="environment" so it opens the rear camera directly on Android/iOS without showing a file picker first. If it currently opens a file picker on mobile, fix it to open the native camera.

Phase 2 — Uploading Screen

While photos upload to POST /api/assessments:

  • Show a full-screen overlay with animated green progress indicator (not just a spinner)
  • Text: "Uploading photos…" then "Analyzing with AI…" (change the message when the analyze call starts)
  • Do NOT allow tapping back — the flow must complete

Phase 3 — Analyzing Screen

Matches prototype screen "Screen 3 — Analyzing". Must show:

  • A scanning animation (CSS pulse/scan line across the equipment photo thumbnail)
  • Text: "AI is identifying your equipment…"
  • Sub-text: "Usually takes 8–15 seconds"
  • A progress indicator showing the stages: Uploading → Reading photos → Identifying model → Checking recalls → Building estimate

Implementation: After the upload completes, immediately call POST /api/assessments/{id}/analyze. This is the Gemini call. It can take 8–20 seconds. Poll GET /api/assessments/{id} every 2 seconds until status === "analyzed", then automatically transition to the results phase.

Phase 4 — Intelligence Results Screen

Matches prototype screen "Screen 4 — Intelligence Results". Must show:

  • Identified Equipment card: Brand + Model + Tonnage + SEER (from ai_equipment_id). Confidence badge (e.g. "94% confidence"). Age estimate.
  • Condition card: Overall condition (CRITICAL / POOR / FAIR / GOOD) with color coding. Red=CRITICAL, Orange=POOR, Yellow=FAIR, Green=GOOD.
  • Issues list: Each issue from ai_issues shows: component name, severity dot (red/orange/yellow), description in plain English (no jargon).
  • Tech Override section: Collapsed by default ("✏ Override AI readings"). Tap to expand fields: Brand, Model, Condition, Year — all editable. Submits PATCH /api/assessments/{id}.
  • Single CTA: "Build Estimate →" button (brand-green, full-width). Calls POST /api/estimates to create the estimate, then navigates to /estimate/{id}.

Error States (all must be handled)

// Every API call must have an error handler that shows a human-readable message
// Never show a blank white screen or a raw JSON error

Upload fails:
  → "Couldn't upload photos. Check your connection and try again."
  → [Retry] button returns to capture phase with photos still selected

Gemini AI times out / returns error:
  → "AI analysis took too long. This sometimes happens on slow connections."
  → [Try again] button re-calls the analyze endpoint
  → [Enter manually] button skips AI and goes to manual equipment entry form

Estimate creation fails:
  → "Couldn't build estimate. The assessment was saved."
  → [Try again] button

All error messages:
  → Brand-red text on light red background card
  → No stack traces, no API error codes visible to the user
  → Always include an action button — never a dead end

Acceptance Criteria

  • On a real mobile device (or Chrome DevTools mobile emulation), tapping the camera button opens the rear camera directly (not a file picker)
  • Uploading 1–3 photos shows the uploading → analyzing progress screens with correct messages
  • The results screen shows equipment ID, condition card, and issues list from the AI response
  • If the API fails at any step, a human-readable error message with a Retry button appears — never a blank screen
  • "Build Estimate →" creates the estimate and navigates to /estimate/{id}
  • All 5 phases match the corresponding screens in ScopeSnap_Prototype_Demo.html

RW-06 Offline Detection & Connection Error Handling
~2 hrs P2 — Field Reliability

Why This Exists

HVAC techs work in basements, utility closets, and rural properties with poor signal. The current app shows a blank white screen or an unhandled promise rejection when the API is unreachable. On a job site, that is a product failure. This WP adds a lightweight offline layer — no service workers, no complex caching — just clear error states and graceful recovery.

Steve Jobs rule: The app must never leave the user stranded. Every dead end must have a door out. "You're offline" is useful. A blank screen is abandonment.

What to Build

1. Global offline banner — create components/OfflineBanner.tsx:

// Add a safeFetch() utility that wraps every API call:
//   - On network error (TypeError: Failed to fetch): throw a typed OfflineError
//   - On 5xx server error: throw a typed ServerError
//   - On 4xx: throw a typed ApiError with the detail message
//   - Never let raw errors bubble to the UI

// Every page that calls the API should catch OfflineError and show:
//   "You're offline. This page will load when your connection returns."
//   + a Retry button that re-calls the fetch

export class OfflineError extends Error { name = "OfflineError" }

export class ApiError extends Error {
  constructor(public status: number, message: string) { super(message) }
  name = "ApiError"
}
