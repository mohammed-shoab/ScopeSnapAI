WP-02 Photo Upload + Vision AI Analysis
~4-6 hrsWeek 1

What to Build

The core AI pipeline: upload 1-5 equipment photos → call Gemini 2.5 Flash with the structured prompt → parse JSON response → store results. This is the single most important piece of the product.

Reference

Tech Spec §03 POST /api/assessments and POST /api/assessments/{id}/analyze
Tech Spec §04 EQUIPMENT_ANALYSIS_PROMPT — use this exact prompt
Tech Spec §01 Request Flow — steps 3-5
Prototype SnapAI_Prototype_Demo.html — Capture → Analyzing → Intelligence Results screens

Deliverables

  1. POST /api/assessments — Accept multipart upload (1-5 images). Compress to 1200px wide server-side (Pillow). Save to LocalStorage. Create assessment record in DB with photo_urls and status='pending'.
  2. POST /api/assessments/{id}/analyze — Load photos from storage. Send to Gemini 2.5 Flash with EQUIPMENT_ANALYSIS_PROMPT from Tech Spec §04. Parse JSON response. Store full AI response in assessments.ai_analysis. Store structured equipment ID in assessments.ai_equipment_id. Store condition in assessments.ai_condition. Store issues in assessments.ai_issues. Store photo annotations in assessment_photos table. Update status to 'analyzed'.
  3. PATCH /api/assessments/{id} — Tech overrides any AI field (brand, model, condition). Store overrides in assessments.tech_overrides JSONB.
  4. Gemini service wrapper: services/vision.py — handles API call, retries (max 2), timeout (30s), error handling (if Gemini returns non-JSON, log and return error). Falls back gracefully if confidence < 50 (returns result but flags as low confidence).

Acceptance Criteria

  • Upload 2 HVAC equipment photos → get back assessment_id within 2 seconds
  • Call /analyze → get structured JSON with brand, model, condition, components, annotations within 15 seconds
  • Response includes photo_annotations with x/y coordinates and color codes
  • ai_analysis stores the complete raw Gemini response (for debugging)
  • ai_equipment_id is structured JSON: {"brand":"...","model":"...","confidence":...}
  • PATCH override works — changing brand from "Carrier" to "Trane" stores in tech_overrides
  • Handles edge cases: blurry photo (low confidence returned), non-HVAC image (error message), Gemini timeout (retry once then return error)

Dependencies

WP-01 must be complete (database, folder structure, storage abstraction).

Test Data

Download 10-20 HVAC equipment photos from Google Images or manufacturer sites. Include: Carrier outdoor unit (model plate visible), Trane furnace, Goodman AC (corroded coil), blurry photo, non-HVAC photo (car engine). Save in test_data/photos/.

THIS IS THE PRODUCT. If this work package produces accurate equipment identification from photos, SnapAI works. If it doesn't, nothing else matters. Spend extra time on prompt tuning here. Test with at least 20 different equipment photos before moving on.
WP-03 Equipment Database + Serial Number Decoding
~4-5 hrsWeek 1

What to Build

Seed the global equipment_models reference table with top 50 HVAC models. Build serial number decoder for all 5 brands. Build the model-matching logic that links a Vision AI result to a known model.

Reference

Tech Spec §07 Equipment Database — brands, models, data sources, serial decode example
Tech Spec §02 equipment_models and equipment_instances tables

Deliverables

  1. Seed script: scripts/seed_equipment_db.py — Populates equipment_models with 50 entries (10 per brand: Carrier, Trane, Lennox, Goodman, Rheem). Each entry has: brand, model_series, model_pattern (regex), equipment_type, seer_rating, tonnage_range, manufacture_years, avg_lifespan_years, known_issues (JSONB), serial_decode_pattern (JSONB).
  2. Serial decoder: services/serial_decoder.py — Given a serial number + brand, returns {"year": 2016, "week": 20}. Implement for all 5 brands using the pattern stored in equipment_models.serial_decode_pattern. Falls back to null if pattern doesn't match.
  3. Model matcher: services/equipment_matcher.py — Given brand + model_number from Vision AI, queries equipment_models using the model_pattern regex. Returns the matching model with lifecycle data, known issues, and recalls. Returns null if no match (model goes into an "unmatched" queue for future DB expansion).
  4. Integration with WP-02: After Vision AI returns results, automatically call model matcher and serial decoder. Enrich assessment response with lifecycle data.

Acceptance Criteria

  • Seed script populates 50 models without errors
  • Query for brand="Carrier" returns 10 models
  • Model matcher: input "Carrier" + "24ACC636A003" → matches to 24ACC6 series → returns lifecycle data (avg lifespan 15 yrs, known coil corrosion issue)
  • Serial decoder: input serial "3516E12345" for Carrier → returns year=2016, week=16
  • Unknown model (e.g., "BrandX" + "ZZZ999") → returns null match, no crash
  • equipment_instances record created/updated when assessment is analyzed at a property

Dependencies

WP-01 (database). Can run in parallel with WP-02 (vision AI) — they integrate at the end.

WEEK 2 — Estimation Engine
WP-04 Pricing Database + Estimate Generation Pipeline
~5-6 hrsWeek 2

What to Build

The estimation engine. Takes AI analysis results + pricing data → generates Good/Better/Best options with line items. Zero AI cost — all deterministic math and DB lookups.

Reference

Tech Spec §05 Estimate Generation Pipeline — all 9 steps, job-type mapping table
Tech Spec §02 estimates, estimate_line_items, pricing_rules tables
Tech Spec §03 POST /api/estimates/generate, PATCH /api/estimates/{id}

Deliverables

  1. Pricing seed script: scripts/seed_pricing.py — Populates pricing_rules with national defaults for common HVAC job types: coil_replacement, full_system, compressor_replacement, refrigerant_recharge, maintenance, repair. Each rule has: parts_cost (min/max/avg), labor_hours (min/max/avg), permit_cost, refrigerant_cost_per_lb.
  2. POST /api/estimates/generate — Takes assessment_id + optional tech adjustments. Runs the 9-step pipeline from Tech Spec §05: determine job types from condition → lookup pricing (cascade: company → region → national) → calculate line items → apply markup → calculate energy savings → build Good/Better/Best options → calculate 5-year total cost. Returns complete estimate with options array.
  3. PATCH /api/estimates/{id} — Tech adjusts line items, markup percentage, or option details. Recalculates totals.
  4. Condition-to-options mapping: Stored as config (not hardcoded). Maps condition combos to 2-3 tier options per Tech Spec §05 table.

Acceptance Criteria

  • For a Carrier AC with moderate coil corrosion: returns 3 options — Clean & Treat (~$450), Replace Coil (~$2,500), New System (~$8,400)
  • Each option has itemized line items (parts, labor, permits, disposal)
  • Markup slider: changing from 35% to 45% recalculates all totals correctly
  • Pricing cascade works: if company pricing exists → use it. If not → use national default
  • 5-year total cost calculated for each option (includes upfront + energy + likely repairs)
  • Energy savings calculated using SEER comparison formula (old SEER vs new SEER)
  • All amounts stored as decimal(10,2) — no floating point rounding errors

Dependencies

WP-01 (database), WP-02 (AI analysis results to feed into estimation).

NO AI IN THIS PIPELINE. If you find yourself calling Gemini or any LLM to calculate pricing, you're doing it wrong. This is SQL queries + arithmetic + template fill. The AI already did its job in WP-02. This step is pure math.
